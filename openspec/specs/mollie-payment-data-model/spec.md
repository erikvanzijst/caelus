# mollie-payment-data-model Specification

## Purpose

Define the mollie_payment table for tracking all Mollie payments, the
MolliePaymentStatus enum, the mollie_customer_id column on the user table,
and the Alembic migration for the Mollie payment schema.

## ADDED Requirements

### Requirement: MolliePaymentORM records every Mollie payment

The system SHALL store Mollie payment records in a `mollie_payment` table. Each record
represents a single payment processed by Mollie, whether created by us (first payment)
or auto-generated by a Mollie subscription (recurring payments).

The table SHALL have these columns:
- `id`: UUID primary key
- `subscription_id`: foreign key to `subscription.id`, NOT NULL
- `mollie_payment_id`: string, UNIQUE, NOT NULL — Mollie's payment identifier (e.g. `tr_xxxxx`)
- `status`: enum, NOT NULL — one of: `open`, `pending`, `authorized`, `paid`, `canceled`, `expired`, `failed`
- `sequence_type`: string, NOT NULL — `first` or `recurring`
- `amount_cents`: integer, NOT NULL — payment amount in euro cents
- `created_at`: datetime, NOT NULL
- `payload`: JSON (JSONB on PostgreSQL) — full Mollie payment response object

All amounts are in EUR. No currency column is needed.

#### Scenario: First payment record created during deployment creation
- **GIVEN** a paid deployment is being created
- **AND** the Mollie API returns payment ID `tr_abc123`
- **WHEN** the deployment creation transaction commits
- **THEN** a `mollie_payment` record exists with:
  - `mollie_payment_id = "tr_abc123"`
  - `status = "open"`
  - `sequence_type = "first"`
  - `amount_cents` matching the plan template's `price_cents`
  - `payload` containing the full Mollie payment response as JSON

#### Scenario: Recurring payment record created by webhook
- **GIVEN** a Mollie subscription auto-generates a payment `tr_rec456`
- **WHEN** the webhook is processed for this payment
- **THEN** a new `mollie_payment` record is inserted with:
  - `mollie_payment_id = "tr_rec456"`
  - `status` matching the Mollie payment's current status
  - `sequence_type = "recurring"`
  - `payload` containing the full Mollie payment response

#### Scenario: Payment status updated by webhook
- **GIVEN** a `mollie_payment` record exists with `status = "open"`
- **WHEN** a webhook arrives and the Mollie API reports `status = "paid"`
- **THEN** the record's `status` is updated to `"paid"`
- **AND** the `payload` is updated with the latest Mollie response

#### Scenario: Unique constraint on mollie_payment_id
- **GIVEN** a `mollie_payment` record exists with `mollie_payment_id = "tr_abc123"`
- **WHEN** another record is inserted with `mollie_payment_id = "tr_abc123"`
- **THEN** the database rejects the insert with a unique constraint violation

### Requirement: Mollie payment status enum

The system SHALL define a `MolliePaymentStatus` enum with these values matching
Mollie's payment lifecycle:

- `open` — payment created, customer has not yet completed it
- `pending` — payment process started but not yet complete (e.g. bank transfer)
- `authorized` — payment authorized but not yet captured
- `paid` — payment completed successfully (terminal)
- `canceled` — payment canceled by customer or merchant (terminal)
- `expired` — payment expired/timed out (terminal)
- `failed` — payment failed and cannot be retried (terminal)

#### Scenario: All Mollie statuses representable
- **WHEN** the `MolliePaymentStatus` enum is inspected
- **THEN** it contains exactly: `open`, `pending`, `authorized`, `paid`, `canceled`, `expired`, `failed`

### Requirement: UserORM has mollie_customer_id

The `user` table SHALL have a `mollie_customer_id` column (nullable string) that
stores the Mollie customer identifier (e.g. `cst_xxxxx`). This links a Caelus user
to their Mollie customer object, enabling recurring payments across all the user's
subscriptions.

A user MAY have at most one Mollie customer. The field is NULL until the user first
creates a paid deployment.

#### Scenario: New user has no Mollie customer
- **WHEN** a new user is created
- **THEN** `mollie_customer_id` is NULL

#### Scenario: Mollie customer ID stored after first paid deployment
- **GIVEN** a user with `mollie_customer_id = NULL`
- **WHEN** the user creates their first paid deployment
- **AND** the payment provider creates a Mollie customer
- **THEN** `user.mollie_customer_id` is updated to the Mollie customer ID (e.g. `cst_xxxxx`)

#### Scenario: Existing Mollie customer reused for subsequent deployments
- **GIVEN** a user with `mollie_customer_id = "cst_abc123"`
- **WHEN** the user creates another paid deployment
- **THEN** no new Mollie customer is created
- **AND** the existing `cst_abc123` customer ID is used for the payment

### Requirement: Payment history is append-only per subscription

The `mollie_payment` table builds up a chronological history of all payments for each
subscription. Records are never deleted. Status updates modify existing records when
the payment ID is known; new records are inserted for auto-generated recurring payments.

#### Scenario: Payment history after three months of a monthly subscription
- **GIVEN** a subscription created 3 months ago on a EUR10/month plan
- **AND** all payments were successful
- **WHEN** the mollie_payment records for that subscription are queried
- **THEN** 4 records exist:
  - 1 with `sequence_type = "first"`, `status = "paid"`, `amount_cents = 1000`
  - 3 with `sequence_type = "recurring"`, `status = "paid"`, `amount_cents = 1000`

#### Scenario: Payment history preserves failures
- **GIVEN** a subscription where month 3's recurring payment failed
- **WHEN** the mollie_payment records are queried
- **THEN** the month 3 record has `status = "failed"`
- **AND** the month 4 record (if it exists) is a separate record

### Requirement: Alembic migration for Mollie payment schema

A single Alembic migration SHALL:
1. Add `mollie_customer_id` (nullable string) to `user` table
2. Add `mollie_subscription_id` (nullable string) to `subscription` table
3. Add `mollie_mandate_id` (nullable string) to `subscription` table
4. Drop `external_ref` from `subscription` table
5. Add `pending` value to `payment_status` enum
6. Create `mollie_payment` table with all columns defined above

No data backfill is needed — existing records are unaffected:
- Existing users have `mollie_customer_id = NULL`
- Existing subscriptions have `mollie_subscription_id = NULL`, `mollie_mandate_id = NULL`
- No existing subscriptions have `payment_status = 'pending'`

#### Scenario: Migration creates mollie_payment table
- **WHEN** the migration is applied
- **THEN** the `mollie_payment` table exists with all specified columns and constraints

#### Scenario: Migration adds Mollie columns to existing tables
- **WHEN** the migration is applied
- **THEN** `user.mollie_customer_id` column exists (nullable string)
- **AND** `subscription.mollie_subscription_id` column exists (nullable string)
- **AND** `subscription.mollie_mandate_id` column exists (nullable string)
- **AND** `subscription.external_ref` column no longer exists

#### Scenario: Existing data unaffected by migration
- **GIVEN** existing users and subscriptions in the database
- **WHEN** the migration is applied
- **THEN** all existing records are preserved
- **AND** new nullable columns default to NULL
